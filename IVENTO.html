  1. Introduction

  1. Purpose

The purpose of the project is to help people to find an easy solution about the budget and elements of their events. “IVENTO” will allow people to find their needed items and see it in 3D view via augmented reality.

The project “IVENTO” will be designed to manage the events in the party. This will take the users information about party events according to their requirement and users can also buy their required product by using “IVENTO” apps as well as on the web.

  1. Document Conventions

‘IVENTO’ is an event management system project where we will arrange different types of event. Customers can order their required product through this website.

  1. Intended Audience and Reading Suggestions

  • Developers can easily understand the project's requirements and where their effort should be targeted. 

  • Tester can use it as a base of their testing strategy and find the most suitable way to test and find bugs 

  • The user can know by this document what our project can do.

  1. Product Scope

“IVENTO” is the platform where users can organize and buy decorating products.  the user can see the product in his/her place using AR. They can choose the party hall or room.

it helps the user to select perfect products for their function in their limited budget. It helps users to make decisions

The goal of the project is to help people by making event organization easy and make out money by selling those products.

  1. Overall Description

  1. Product Perspective

“INVENTO” is a very useful platform for event organization and buying decorated products. It is a free app and web based service.

  1. Product Functions

  • Maintain records for customers

       . A customer can either be a member or non-member

       . A customer has a username, password, address and phone no

       . Anyone may sign up for a account

  •  Maintain data associated with the inventory

       . A product has a title and price

       . The inventory also keeps the track of the quantity of the product

  •  Allow user to log in and log out of the system
  • Show list of available function 
  •  Show list of available product corresponding to the function
  •  Allow customer to become a member
  •  Allow customer to select an event and a venue for doing their events
  • Allow customers to see the products in 3d view by AR
  • Product inventory system
  • Allow customer to buy individual product

  1. System Feature 1

4.1.1        Description and Priority

The project ‘INVENTO’ where we will manage the event at a party with their conditions and requirements. Here, customers can buy different products according to their requirements. 

4.1.2        Stimulus/Response Sequences

                       Stimulus: User select an event 

                       Response: System shows user corresponding products 

                       Stimulus: User select a product 

                       Response: System show the price and a option of 3D view

                      Stimulus: User select the 3D view option 

                      Response: System show it using AR

4.1.3        Functional Requirements

TITLE: User registration

DESC: After the user has downloaded the software, then he/she is able to register                                  through the application. The user must provide user-name, password, phone no and address.

TITLE: User log-in

DESC: After the user has registered, then the user should be able to log in to the application. The log-in information is stored on the database and in future the user will be logged in automatically.

TITLE: Retrieve password

DESC: After user has registered, then user is able to retrieve his/her password by email

Feature: Admin

DESC: In order to keep track of the order an administrator is able to manage the IVENTO owners.

TITLE: Select event

DESC: User can select an event

TITLE: Search product

DESC: By using this software user can search their desire product and buy it

TITLE: View product in AR

DESC: By using IVENTO app user can see some 3D view of some decoration product

  1. Performance Requirements

Although the system is a simple one a user who is able to understand simple mobile processes is needed to run the system. The user will be the person to enter the data needed into the system, thus a user needs to also be efficient to utilize fully the benefits that can be provided by the system.

It also should use less memory so memory management should be done perfectly. When doing a large processing job system needs to allow doing it firstly. 

To make IVENTO software always up-to-date with new features, developers need to always update it and fix its bugs.

  1. Safety Requirements

System will not break any governmental and ethical law.

We respect the privacy of our clients.

Different information is entered into the system such as information about the customers and sellers. Mismanagement of information might cause user dissatisfaction that will eventually lead to profit loss, only because of mistakes on giving information.

  1. Security Requirements

  • Unauthorized access to the system and its data is not allowed.

  • The user has respective accounts with password that enable only the user to log into the system

  • Password are required so that no one else can access the account
  • Ensure the integrity of the system from malicious demand 

  • Sign-off Button

  • Failed Login Attempts

  1. Software Quality Attributes

. Correct information must be entered into the system to prevent mismanaged conflicts to occur.

.  This will make the information provided by the system to be reliable and useful. 

. We are also concerned about the error. In case an error occurs, changes may be immediately affected provided the user notices the error.

. The users of this system need to understand the basics about phones so use of this system will be easy for them.

. The performance quality is concerned with response time  

  1. Business Rules

. No user is allowed to see other user information.

. One user can have only one account.

. All search history will be save in database

. Every user should have treated equally
